Tier 1 — go deep

Cryptography

The five-way distinction, the symmetric/asymmetric pattern reused everywhere, PKI trust, forward secrecy, and where post-quantum crypto stands in 2026.

Five-way distinction

This is a great structuring device for any crypto answer — lead with it, then let the interviewer pull you deeper.

Encryption
Secrecy. Reversible with a key.
Encoding
Compatibility. Reversible, no key — Base64, URL encoding.
Hashing
Integrity. One-way, fixed-length fingerprint.
Obfuscation
Hindrance. Reversible, no formal guarantee.
Signing
Authenticity. Hash + private-key stamp, verified with the public key.

Symmetric vs asymmetric — know the why, not just the definitions

Real protocols (TLS, PGP, Signal) use asymmetric crypto only to establish a symmetric session key, then switch to symmetric for the actual payload. This "asymmetric for exchange, symmetric for bulk" pattern is the single most reused idea in applied cryptography — recognize it everywhere.

AES-GCM specifically

An AEAD (authenticated encryption with associated data) cipher — it provides confidentiality and integrity in one primitive, which is why it replaced separate encrypt-then-MAC constructions in most modern protocols.

Nonce reuse is catastrophic
GCM's security depends on never reusing a nonce with the same key. Reuse leaks the XOR of two plaintexts and can recover the authentication key.

PKI and trust

A CA signs a certificate binding a public key to an identity; browsers/OSes ship a root store of CAs they trust; trust is transitive down the chain. DigiNotar (2011) is the canonical failure case — a CA was compromised and issued fraudulent certificates (including for *.google.com), used to intercept Iranian Gmail users' traffic.

Certificate Transparency exists precisely because DigiNotar-style silent mis-issuance was undetectable before public, append-only, auditable logs of every issued certificate.

Forward secrecy, concretely

In a Signal/WhatsApp-style protocol, the Double Ratchet derives a new symmetric key per message from a continuously advancing key chain, so compromising today's key doesn't expose yesterday's or tomorrow's messages. This is the practical extension of the ECDHE idea in TLS — same principle (ephemeral keys), applied at message granularity instead of session granularity. Heartbleed is the best real-world example for explaining forward secrecy's actual guarantee precisely.

Entropy

PRNGs need a real entropy source to seed from. Entropy buffer draining (e.g. a freshly booted VM/container with no accumulated hardware entropy) has caused real key-generation weaknesses — embedded devices have shared RSA keys because they generated keys before the entropy pool filled.

2025–2026 update: post-quantum cryptography is now mandated, not researched

NIST finalized three PQC standards in August 2024:

FIPS 203
ML-KEM — key encapsulation
FIPS 204
ML-DSA — digital signatures (default)
FIPS 205
SLH-DSA — hash-based backup

ML-DSA (lattice-based) is NIST's recommended default replacement for RSA/ECDSA signing. SLH-DSA is kept as a structurally independent backup in case lattice assumptions are ever broken. The NSA's CNSA 2.0 sets 2030 as the mandatory PQC migration deadline for National Security Systems, pulling forward migration timelines industry-wide.

Practice questions

  • Why is nonce reuse catastrophic in AES-GCM but a different failure mode in AES-CBC?
  • What happens when a trusted CA is compromised, and how would you design around that risk?
  • Why does a hash-based signature scheme (SLH-DSA) exist as a backup to lattice-based ML-DSA?
