Tier 1 — go deep

Web application security

Root cause and mechanism for every classic web vulnerability — not just the label. Root cause is what signals depth in an interview.

Same-origin policy & CORS

SOP is the default-deny browser boundary (scheme + host + port must match). CORS is an explicit, server-declared relaxation of SOP via response headers. For "unsafe" requests (custom headers, non-simple methods, credentialed requests) the browser sends a preflight OPTIONS request first, specifically to prevent a malicious page from silently triggering state-changing cross-origin requests.

CSRF vs XSS — the distinction interviewers actually probe for

CSRF exploits the browser's automatic inclusion of ambient credentials (cookies) on requests to a site the victim is already authenticated to — the attacker never reads the response, they just cause a side effect. Defenses: CSRF tokens, the SameSite cookie attribute (browsers defaulting to Lax has made classic CSRF much rarer), and checking Origin/Referer headers.

Reflected XSS
Payload round-trips through a request parameter, immediately rendered — needs social engineering to deliver the malicious link.
Persistent / stored XSS
Payload saved server-side (e.g. a comment field), executes for every subsequent viewer — highest impact.
DOM-based XSS
The vulnerability is entirely client-side JS taking untrusted data (e.g. location.hash) and writing it into the DOM via innerHTML, without the payload ever touching the server — server-side sanitization can't catch it.

Root-cause defense: context-aware output encoding plus Content-Security-Policy as defense in depth, not primary defense.

SQL injection

Union-based
Leverage UNION SELECT to pull data from other tables through a vulnerable query.
Error-based
Force DB errors that leak schema info in the error message.
Blind (boolean / time-based)
No visible output — infer true/false one bit at a time via response differences or SLEEP() timing.

Root cause is always the same: untrusted input concatenated into a query string instead of passed as a parameter. Parameterized queries close the entire class because the query structure is compiled before user input is bound, so input can never be reinterpreted as SQL syntax.

SSRF

The server itself is tricked into making a request to an attacker-chosen destination — classically used to reach internal-only services like cloud metadata endpoints. See the Capital One breach for the full real-world chain from SSRF to 106 million stolen records. Defenses: allowlist outbound destinations, disable redirects on server-side fetches, network-segment the fetching service away from sensitive internal endpoints.

Directory traversal / LFI / RFI

../../etc/passwd-style path manipulation. Root cause is unsanitized file-path construction from user input; defense is canonicalizing the resolved path and checking it stays within an allowed root.

2025–2026 addition: prompt injection

The OWASP Top 10 for LLM Applications (2025) is now a live framework, and Google is shipping LLM-integrated products broadly. Prompt injection holds the #1 spot for the second edition running — an attacker embeds instructions in content the model will ingest to override its intended behavior.

Same root cause as SQLi and XSS
Untrusted data and trusted instructions share the same channel, and the model can't reliably distinguish them. New 2025 categories worth knowing: excessive agency (an agent with more tool permissions than a task needs, directly analogous to least-privilege violations elsewhere), system prompt leakage, and vector/embedding weaknesses in RAG pipelines. If a design question involves an AI component, raise prompt injection unprompted — it's a strong signal of current awareness.

Practice questions

  • Why does a DOM-based XSS bug evade server-side sanitization entirely?
  • What's the classic cloud escalation path from an SSRF bug?
  • Why do parameterized queries fully close SQL injection rather than just filtering it?
