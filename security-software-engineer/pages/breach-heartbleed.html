Breach case study2014

Heartbleed

CVE-2014-0160 — the single most-cited real-world justification for memory-safe languages in security infrastructure.

A Google engineer found it, and it's Google's own push toward memory-safe languages that this bug is most often cited to justify.

The mechanism

The TLS heartbeat extension (RFC 6520) exists to keep a connection alive and confirm the peer is still there without a full renegotiation. The client sends a payload and a length field; the server is supposed to echo back exactly that payload. OpenSSL's implementation read the 2-byte length field straight out of attacker-controlled input, then called memcpy(bp, pl, payload), copying payload bytes starting at the pointer to the client's data — without checking that payload actually matched the number of bytes the client had sent.

A client could send one byte of real data but claim the length was 65535, and the server would memcpy 64KB starting at that one-byte pointer, reading straight past the end of the real payload into whatever else happened to be sitting in adjacent heap memory, then hand all of it back in the response.

Why private keys specifically
OpenSSL's heap arena gets reused constantly across connections — recently freed buffers holding other users' request data, session keys, and, if the memory layout happened to line up, the server's own long-term private key structure. It's probabilistic per request, which is why exploitation looked like repeated automated probing. Cloudflare ran an open challenge in April 2014 daring researchers to pull a private key out of a deliberately vulnerable nginx box — two independent researchers succeeded within hours.

Read vs write, and why that distinction matters

This is an out-of-bounds read, not a write — no return address or control data got corrupted, so there's no code execution here, only information disclosure. Not every memory-safety bug is RCE-flavored: over-reads leak confidentiality, over-writes threaten integrity and control flow, and they call for different mitigations.

Two years of silence

The code had shipped in OpenSSL 1.0.1 back in March 2012, and sat there for over two years before a Google security engineer found it during a line-by-line audit of OpenSSL prompted by two earlier SSL bugs. This bug produced no crash, no corrupted state, nothing that would show up in a crash-monitoring dashboard — there is no way to retroactively prove whether it was quietly exploited by a well-resourced adversary before public disclosure in April 2014. A vulnerability class that's silent by construction is a much harder detection problem than one that trips an alert.

Forward secrecy nuance

If a server used ephemeral ECDHE and its long-term private key later leaked via Heartbleed, past recorded sessions were still safe — forward secrecy's whole point is that stealing the long-term key after the fact doesn't unlock previously captured traffic. What it doesn't protect is the live session running at the moment of compromise, since the plaintext or session key for that connection could itself be sitting in the same leaked memory. See the full explanation on the networking & TLS page.

The throughline to today

Memory-safe languages
A missing bounds check in hand-written C is structurally impossible to write the same way in Rust — slice access is bounds-checked at runtime and panics rather than reading adjacent memory. That's precisely the class of bug (silent over-read, not RCE via corruption) that Google's Rust-in-Android push is built to eliminate by construction rather than catch through review.
