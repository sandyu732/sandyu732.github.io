Tier 1 — go deep

Authentication & identity

Cookies and sessions, OAuth and OIDC, Kerberos and its named attacks, TPM, MFA comparison, and why passkeys changed the calculus in 2025.

Cookies vs sessions

Mark auth cookies HttpOnly (blocks JS access, mitigating XSS-based theft), Secure (HTTPS-only transmission), SameSite (mitigates CSRF). Server-side sessions store state keyed by an opaque session ID — more revocable than a self-contained token.

OAuth 2.0 authorization code flow, with PKCE

OAuth 2.0 is an authorization protocol, not authentication — it grants a bearer token scoped to specific permissions, delegated by a resource owner to a client, without sharing the owner's credentials with that client. PKCE (now standard even for public clients) proves the app exchanging the code is the same one that started the flow, preventing a stolen authorization code from being redeemed by an attacker.

OpenID Connect sits on top of OAuth 2.0 to add actual authentication (the ID token, a signed JWT asserting identity). A common interview trap is conflating OAuth with authentication when it's fundamentally an authorization framework.

SAML vs OIDC: SAML is XML-based, enterprise/SSO-heavy, older. OIDC is JSON/JWT-based, REST-friendly, dominant for consumer and modern app SSO. Both solve the same federated-identity problem with different wire formats and eras of design.

Kerberos

Ticket-based auth using a trusted third party, the Key Distribution Center (KDC). The two-ticket flow (TGT from the Authentication Server, then a service ticket from the Ticket Granting Server) avoids repeatedly transmitting a password.

Golden ticket
Forge a TGT by stealing the KDC's krbtgt account hash — grants domain-wide access. Used in the SolarWinds intrusion for lateral movement.
Silver ticket
Forge a service ticket using a stolen service-account hash — narrower, per-service access, and harder to detect since it doesn't touch the KDC.
Pass-the-hash
Authenticate using a stolen NTLM hash directly, without ever knowing the plaintext password — works because NTLM auth only ever needs the hash.

TPM

Hardware-backed secure storage for keys/certs that never leaves the chip in plaintext — the root of trust for disk encryption (BitLocker) and platform attestation.

MFA comparison — know the threat model each factor defeats

FactorWhat breaks it
SMS OTPWeakest — vulnerable to SIM-swap.
TOTP appsBetter, but still phishable via real-time relay / adversary-in-the-middle proxies.
Push notificationsVulnerable to "MFA fatigue" — spamming approval requests until the user taps accept. See the Uber breach.
U2F / FIDO2 hardware keysPhishing-resistant by design — the cryptographic challenge is bound to the origin, so a phishing site cannot complete the handshake.

2025–2026 update: passkeys crossed from novelty to default

Passkeys (FIDO2/WebAuthn credentials, often synced via a platform keychain) reached mainstream enterprise deployment in 2025; roughly half of surveyed users now have at least one enabled.

NIST SP 800-63-4 (July 2025)
Now recognizes synced passkeys as meeting AAL2, and requires phishing-resistant authenticators at that assurance level rather than merely recommending them — a meaningful regulatory shift. A passkey's private key never leaves the device/secure enclave; authentication is a signed, origin-bound challenge, which is exactly why it resists phishing the same way U2F does.

Identity in cloud / service context

Service accounts are non-human identities used for automation and should carry least-privilege scopes. Exported service-account keys are a major attacker target because they're long-lived, portable bearer credentials with no built-in rotation. Workload identity federation (short-lived, attested tokens instead of long-lived exported keys) is the modern mitigation — the right answer if asked how to avoid this risk.

Practice questions

  • Why is OAuth 2.0 not, by itself, an authentication protocol?
  • What problem does PKCE solve for public clients like mobile apps?
  • Why does pass-the-hash work against NTLM but not against a properly implemented challenge-response scheme using salted, non-replayable proofs?
