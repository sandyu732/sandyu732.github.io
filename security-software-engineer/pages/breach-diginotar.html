DigiNotar CA breach

How a compromised certificate authority silently undermined the trust model behind every HTTPS connection on the internet.

The attacker went by ComodoHacker — the same identity behind an earlier, partially failed attempt against a Comodo registration authority months earlier. What makes DigiNotar the canonical PKI failure example isn't just that a CA got hacked; it's that once inside, the attacker had almost no resistance.

Why it worked

DigiNotar's network was segmented in name only. The "Secure-net" segment holding the actual CA signing servers was supposed to be unreachable from the internet, but the attacker tunneled through already-compromised systems to reach it about two weeks after the initial breach. Once there, a single admin account had access to all eight certificate servers — no separation between them, several systems had no antivirus, and there was no central logging to notice any of this happening.

The real lesson
PKI's security model assumes the CA's signing infrastructure is a hardened, monitored, least-privilege environment, and DigiNotar's wasn't. The cryptography around certificate issuance was never broken; the operational security around who could invoke it was.

The trust model failure

Any of the hundreds of CAs a browser trusts by default can, if compromised, mint a certificate for any domain, and every other client has no way to tell it apart from the real one just by checking the chain and signature. Verification up the chain (leaf → intermediate → root) only proves a certificate was signed by some trusted CA — it says nothing about whether that CA should have issued that particular certificate.

What actually caught it was independent of PKI entirely: Google had hardcoded, or "pinned," what Google's real certificate looked like inside Chrome. When Chrome encountered the rogue one it refused it outright, and a confused user posting about the error on a help forum unraveled the whole thing. Revocation lists and OCSP wouldn't have helped — those only work once someone already suspects a problem and checks; pinning caught a problem nobody yet knew existed.

What changed because of this

Certificate Transparency exists specifically because of this incident (and the earlier Comodo one). CT requires certificates to be logged in public, append-only, cryptographically verifiable logs before browsers will trust them — a CA silently mis-issuing a certificate for google.com is now detectable within hours by monitoring the logs. Chrome has required CT log inclusion (via embedded SCTs) for all publicly trusted certificates since 2018.

Pinning (and its short-lived successor HPKP) was the stopgap before CT matured, and has largely been retired in favor of CT precisely because pinning is brittle — get it wrong and you lock legitimate users out of your own site. Browsers and OS vendors permanently removed DigiNotar from their root stores, and the company was bankrupt within about a month.

Interview framing

"What happens when a CA is compromised, and how would you design around that risk?"
Don't rely on the trust chain alone being valid — assume any single CA can fail, and layer independent detection on top, which is exactly what CT and monitoring services (crt.sh, cert transparency alerting) do today for any domain you care about.
