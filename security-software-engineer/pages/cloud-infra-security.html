Tier 2 — working knowledge

Cloud & infrastructure security

Isolation boundaries, zero trust, cloud lateral movement, supply chain attacks, and memory safety as an infrastructure strategy.

Isolation boundaries

Containers share the host kernel — a weaker isolation boundary than a hypervisor, which is why container escapes matter more than VM escapes in threat modeling. Common escape vectors: shared kernel vulnerabilities, misconfigured privileged containers, host-path/volume mounts exposing the host filesystem.

BeyondCorp / zero trust

The core inversion: trust the device and identity, not the network. No more privileged internal network — every request is authenticated and authorized regardless of source, replacing perimeter VPN trust.

2025–2026 update: "Beyond Zero"
Google's evolution of BeyondCorp extends the identity-centric approach with real-time, per-action reasoning, because the original assumptions (accessors are human, actions happen at human speed, the application is the right trust boundary) break down when AI agents can issue orders of magnitude more requests than a human. The trust boundary shifts from "the application" to "the specific action on a specific piece of data, evaluated in the moment," not reviewed after the fact.

Lateral movement & privilege escalation in cloud

Exported/over-scoped service account keys, IAM role chaining (a compromised low-privilege identity assumes a more privileged role due to a misconfigured trust policy), and — very current — workload-identity misconfigurations letting a compromised pod assume cloud IAM roles far beyond what the workload needs. See the Capital One breach for the canonical example: an over-scoped role turned one SSRF bug into 106 million stolen records. Container-based lateral movement and service-account token theft are both active, not historical, risk classes in current cloud environments.

Supply chain security

Two case studies worth knowing cold:

SolarWinds / Sunburst (2020)
A trojanized update to Orion software, signed and distributed through the legitimate build pipeline, then Kerberos golden tickets used for lateral movement inside victim networks. Full write-up on the breach page.
xz-utils backdoor (2024)
A two-year social-engineering campaign earned commit access to a widely-depended-upon compression library, then hid a backdoor in the build system targeting sshd. Caught by chance, not process. Full write-up on the breach page.

Both incidents show a compromise of the build/trust pipeline, not the source code review process. Lessons: build-system/CI supply chain is as much an attack surface as source code; maintainer burnout/succession in critical open-source infrastructure is a systemic risk; reproducible builds and provenance attestation (in-toto, SLSA) are the direct mitigations pushed since.

Memory safety as an infrastructure-security strategy

Reduction in vuln density, Rust vs equivalent C/C++

Google's push to migrate Android (and expanding into Chromium parsers, firmware, and more) to Rust has driven this shift. Good example to cite if asked "how would you reduce a class of vulnerabilities at scale" — the answer isn't "review harder," it's "change the language so the bug class becomes structurally rare."

The nuance
Rust doesn't eliminate all memory-safety-adjacent risk. Google's own near-miss with a buffer overflow in a Rust image-decoding component (CrabbyAVIF) is worth knowing: safety is a spectrum, and unsafe blocks / logic errors still exist. See the Heartbleed page for the C-era bug this strategy is designed to prevent.
