Capital One SSRF breach

The textbook "walk me through a breach" story — SSRF, cloud IAM, and detection failure in one incident.

Paige Thompson, a former AWS employee, pulled this off in March 2019; it wasn't discovered until July, after she posted about it publicly. Every link in the chain above is a distinct, nameable security concept.

The confused deputy

The root vulnerability was SSRF in Capital One's ModSecurity WAF — a misconfigured rule let an attacker-supplied request get proxied server-side rather than validated. That's mechanically identical to the SSRF you'd write in the coding-practice section: untrusted input controls where the server sends a request. What made it catastrophic was where that forwarded request could go: 169.254.169.254, the EC2 instance metadata service, reachable only from inside the instance — and trusted with zero authentication under IMDSv1. Any process on the box, including one an attacker tricked the WAF into impersonating, could just GET a temporary IAM key.

This is a confused deputy problem
The WAF had legitimate permissions on Capital One's behalf, and the SSRF let an outsider borrow its identity without ever needing a password.

The second failure: least privilege violated

The role attached to that WAF instance could list and read across roughly 700 S3 buckets — vastly more than a web application firewall has any business touching. If that role had been correctly scoped, the stolen credentials would have been nearly worthless. Instead, Thompson ran S3 sync commands bucket after bucket, pulling data on about 106 million people, including roughly 140,000 Social Security numbers and 80,000 linked bank account numbers. Capital One paid an $80M OCC penalty plus a $190M class-action settlement.

The detection gap

The exfiltration ran for months before anyone at Capital One noticed — it surfaced only because Thompson bragged about it and a third party tipped Capital One off. A role suddenly issuing high-volume ListBucket/GetObject calls across hundreds of buckets it doesn't normally touch is a strong anomaly signal; see the detection & IR page for how you'd design monitoring to catch this kind of misuse via baseline correlation, not just static policy review.

How IMDSv2 fixed this class of bug

IMDSv2 requires first sending a PUT request with a custom header to obtain a session token before any metadata GET will succeed, and it defaults to a hop limit of 1. Most SSRF bugs only give an attacker control over a URL fetched via GET — they can't easily inject a custom header or switch the HTTP verb, and even if the vulnerable service is itself a proxy, hop-limit 1 means the token request can't survive being relayed through it. "Authentication was implicit and free" became "authentication requires a request shape a generic SSRF can't produce."

Interview framing

The throughline to state explicitly
SSRF isn't just a web bug — it's dangerous in direct proportion to what the vulnerable service is trusted to reach, and in cloud environments, that's almost always the metadata endpoint, and therefore your entire IAM posture.
