2. The classifier, with structured output

Start with a mock — a few if statements standing in for the model. This is deliberate: it proves your pipeline plumbing (prompt → parse → schema) works before you spend a single API call on it, and it means step 3's evaluation harness is testable immediately.

import json

PROMPT = '''You are a security code reviewer. Classify this snippet.
Respond with ONLY valid JSON matching this schema:
{{"vulnerability": "sqli|ssrf|xss|hardcoded_secret|none", "confidence": 0.0-1.0, "reason": "one sentence"}}

Code:
{code}
'''

def classify_snippet(code, call_llm):
    raw = call_llm(PROMPT.format(code=code))
    try:
        return json.loads(raw)
    except json.JSONDecodeError:
        # never let a malformed response silently pass as "safe" downstream
        return {"vulnerability": "parse_error", "confidence": 0.0, "reason": raw[:80]}

# Start here with zero API cost — a rule-based mock proves the pipeline
# shape works before you spend a token on a real model:
def mock_call_llm(prompt):
    code = prompt.split("Code:\n")[-1]
    if "SELECT" in code and "{" in code:
        return '{"vulnerability": "sqli", "confidence": 0.9, "reason": "f-string in SQL"}'
    if "requests.get(user" in code:
        return '{"vulnerability": "ssrf", "confidence": 0.8, "reason": "unvalidated outbound URL"}'
    return '{"vulnerability": "none", "confidence": 0.95, "reason": "no pattern matched"}'

Why the try/except matters
A model that returns malformed JSON is not a rare edge case in production — it's a Tuesday. Treat a parse failure as its own outcome, never as a silent "safe." This single habit is a strong signal of production ML experience.

3. The evaluation harness

Run it against your mock classifier first. You should get a real precision/recall number, even though there's no real model yet — that's the point. The harness doesn't care what's behind call_llm.

def evaluate(samples, call_llm):
    tp = fp = fn = tn = 0
    for s in samples:
        pred = classify_snippet(s["code"], call_llm)
        predicted_positive = pred["vulnerability"] not in ("none", "parse_error")
        actual_positive = s["label"] != "safe"
        if predicted_positive and actual_positive: tp += 1
        elif predicted_positive and not actual_positive: fp += 1
        elif not predicted_positive and actual_positive: fn += 1
        else: tn += 1

    precision = tp / (tp + fp) if (tp + fp) else float("nan")
    recall = tp / (tp + fn) if (tp + fn) else float("nan")
    print(f"precision={precision:.2f}  recall={recall:.2f}  "
          f"tp={tp} fp={fp} fn={fn} tn={tn}")

evaluate(SAMPLES, mock_call_llm)

4. The guardrail

This is the highest-leverage 15 minutes of the whole exercise.

def route(prediction, blast_radius="high"):
    """blast_radius: how bad it is if an auto-fix here is wrong."""
    if prediction["vulnerability"] in ("none", "parse_error"):
        return "no_action"
    if prediction["confidence"] >= 0.85 and blast_radius == "low":
        return "auto_fix"          # only for reversible, low-stakes changes
    return "human_review_queue"    # everything else waits for a person

# This one function is the entire idea behind "automated guardrails to
# mitigate risk" from the job description. Everything upstream of it can
# be wrong sometimes — this is the layer that decides how much that matters.
